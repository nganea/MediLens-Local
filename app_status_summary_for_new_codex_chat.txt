Core app
--------
The app is called MediLens Local: Medicine Label Helper.

It is a local-first Gradio web app that helps read medicine labels from an
uploaded or captured image. It extracts text from the image, matches the label
against a local medicine database, and displays:

- Medicine match
- Medicine use
- Safety warning
- Source URL
- Extracted OCR text in Technical details

The app does not call cloud APIs.

Local models
------------
The app is designed to use two local llama.cpp servers.

Tiny Aya Global:
- Purpose: rewrite/translate medicine explanations and safety wording.
- Default URL: http://127.0.0.1:8080/v1/chat/completions
- Model ref: CohereLabs/tiny-aya-global-GGUF:Q4_K_M

MiniCPM-V 4.6:
- Purpose: vision OCR for medicine label images.
- Default URL: http://127.0.0.1:8081/v1/chat/completions
- Model ref: openbmb/MiniCPM-V-4.6-gguf:Q4_K_M

Model startup behavior
----------------------
The desktop app no longer starts local AI models just because the webpage loads.

Instead:
- When the user uploads an image or types a medicine name, the app checks/starts
  local model servers.
- While models are being checked or started, the Search button is disabled and
  the text under it says "Loading AI models...".
- If both model ports are already reachable, the app skips the loading state.
- The user can also press "Check/retry local model servers" in Technical details.

GPU offload behavior
--------------------
Set this before launching the app to ask llama.cpp to offload layers to GPU:

export MEDILENS_LLAMA_GPU_LAYERS=99

PowerShell equivalent:

$env:MEDILENS_LLAMA_GPU_LAYERS="99"

When set to a positive integer, the app starts background `llama-server` with:

-ngl <value>

For example, `MEDILENS_LLAMA_GPU_LAYERS=99` adds `-ngl 99`. This means "try to
offload up to 99 layers"; in practice it is a common way to ask llama.cpp to
offload as much as possible. It still depends on the user's GPU, VRAM, and
whether their llama.cpp build supports GPU.

Manual model startup commands
-----------------------------
CPU:

llama-server -hf CohereLabs/tiny-aya-global-GGUF:Q4_K_M --port 8080
llama-server -hf openbmb/MiniCPM-V-4.6-gguf:Q4_K_M --port 8081

GPU:

llama-server -hf CohereLabs/tiny-aya-global-GGUF:Q4_K_M --port 8080 -ngl 99
llama-server -hf openbmb/MiniCPM-V-4.6-gguf:Q4_K_M --port 8081 -ngl 99

Main desktop workflow
---------------------
1. Upload/capture a medicine label image, or type a medicine name.
2. Select the language.
3. If needed, the app checks/starts the local AI model servers.
4. Click Search.
5. MiniCPM-V and Tesseract work together to read the image.
6. The app matches OCR/manual text against medicines.csv.
7. Tiny Aya optionally rewrites/translates the result.
8. The app displays medicine match, medicine use, safety warning, source URL,
   and OCR details.

OCR behavior
------------
MiniCPM-V 4.6 and Tesseract are enabled together by default.

Tesseract is useful for debugging and fallback text, but if MiniCPM-V cannot be
reached or cannot read the image, the app should not trust Tesseract-only OCR
for medicine matching because it can produce incorrect medicine names.

If MiniCPM-V times out:
- The webpage stays open.
- Medicine match shows N/A.
- Medicine use asks the user to type the medicine name or label text and search
  again.
- Safety warning shows N/A.
- The manual label field is highlighted.

Translation behavior
--------------------
Tiny Aya is used to make medicine use and safety warning easier to read and to
translate them.

Desired explanation style:
- Like writing for a 14-15 year old.
- Maximum 3 short sentences.
- Non-repeating.
- No dosage instructions.
- Avoid unnecessary medicine form details unless clinically necessary.

Safety warning style:
- Avoid repeating generic and medicine-specific warnings.
- Use "patient information leaflet" rather than "patient leaflet".
- Do not tell the user that the medicine is safe for them personally.

Safety position
---------------
The app is informational only.

It should not:
- Provide dosage instructions.
- Tell a user to take a medicine.
- Confirm that a medicine is safe for a specific person.
- Replace a pharmacist, doctor, or other qualified healthcare professional.

It should:
- Explain common medicine uses.
- Show source URL.
- Encourage checking with a pharmacist or doctor if unsure.
- Use N/A when the app cannot reliably identify or process the medicine.

Reachy Mini desktop integration
===============================
The desktop app now has a Reachy Mini control in Technical details.

Button:
- Start Reachy Mini MediLens
- Stop Reachy Mini MediLens

Before starting, the status box reminds the user to:
- Turn on Reachy Mini at reachy-mini.local:8000.
- Make sure all robot applications are off.
- Allow Windows Firewall if it asks for permission.

The Reachy button does nothing on page load. It only starts after a user presses
the button.

When Start is pressed:
- The desktop app clears these fields:
  - Photo
  - Medicine name
  - Medicine match
  - Medicine use
  - Safety warning
  - Source
  - Extracted OCR text
- The local robot service starts if needed.
- The hands-free Reachy app starts in the background.
- The app uses Reachy's own microphone by default:
  --mic reachy
- The app writes logs/status/result files under robot/logs/.

Reachy spoken prompt
--------------------
The English listening prompt is now:

"I am listening. When you are ready ask me about a medicine."

The English goodbye phrase is:

"Okay, I'll stop now. Goodbye."

Graceful Reachy shutdown
------------------------
When the desktop Stop button is pressed:
- app.py writes robot/logs/reachy_stop_requested.txt.
- robot/reachy_mini_app.py notices the file.
- Reachy says the goodbye phrase.
- Reachy lowers antennas.
- The Reachy process exits.
- app.py stops the local robot service.
- If the Reachy app does not exit within 12 seconds, app.py force-stops it as a
  fallback.

Reachy run command equivalent
-----------------------------
The desktop button effectively runs:

py -3 robot/reachy_mini_app.py --use-reachy --listen --language auto \
  --orientation-mode "Normal first" --max-vision-attempts 1 \
  --service-url http://127.0.0.1:8765 --reachy-host reachy-mini.local \
  --reachy-port 8000 --timeout 90 --mic reachy --show-result \
  --status-file robot/logs/reachy_status.txt \
  --result-file robot/logs/reachy_result.json \
  --stop-file robot/logs/reachy_stop_requested.txt

Reachy hands-free conversation flow
-----------------------------------
1. Loads + calibrates the mic.
2. Reachy waggles antennas and says the listening prompt.
3. Wake cue = Reachy name + a medicine word.
4. Reachy says Okay, captures the label, does the thinking motion, identifies
   the medicine, and speaks the answer in the same language it detected.
5. It keeps listening for the next question.
6. Say "Reachy stop" or press Stop Reachy Mini MediLens to end.

Known harmless startup noise
----------------------------
- "No Reachy Mini Audio USB device found!" can appear from the client mic-capture
  path; robot playback via play_sound can still work.
- WebRTC "Signalling error: receiver is gone" can appear on exit.
- phonemizer "words count mismatch" warnings from Piper can appear.

Performance notes
-----------------
- MiniCPM-V image OCR is the slowest step.
- `MEDILENS_LLAMA_GPU_LAYERS=99` can speed model inference if llama.cpp is built
  with GPU support and the GPU has enough VRAM.
- Desktop upload uses Normal first orientation, but defaults to 2 attempts per
  image orientation for caution.
- Reachy uses Normal first and 1 attempt for speed.
- Reachy status polling is every 3 seconds and queue=False so it should not slow
  desktop uploads.
- If Reachy and desktop both call MiniCPM at the same time, they may compete for
  the same model server/GPU.

The Space runs the FULL app.py (same design as the desktop app), not a separate

  cp app.py space/app.py
- The Space must use Gradio sdk_version that matches the app (6.18.0). An older
  pin (5.9.1) caused an "Error: No API found" on button click.

Space-aware behavior (no GPU on a free Space):
- app.py sets IS_HOSTED = bool(os.getenv("SPACE_ID")).
- When hosted: an amber "Hosted demo (no GPU)" banner is shown; the two local
  model checkboxes default to OFF; the Reachy status polling timer is skipped.
- What works live on the Space: typed medicine lookup against medicines.csv and
  offline multilingual TEMPLATE explanations in all 6 languages.
- What does NOT run on the Space (no GPU): MiniCPM-V image OCR, Tiny Aya
  AI rewrite/translation, and Reachy Mini. These are shown in the demo video.
- Hosted safety warning: because Tiny Aya cannot translate the free-text
  medicine-specific warning without a GPU, when hosted + AI model off +
  non-English, the safety box shows the fully translated GENERAL safety advice
  instead of a mixed English/translated box. Desktop behavior is unchanged.

Known cosmetic notes on the Space:
- A one-time page "flash" on load comes from the forced-light-theme URL redirect
  (?__theme=light). Intentionally kept so the app always looks light/branded.
- Image upload does not read labels on the Space (MiniCPM-V needs a GPU); this
  is expected and explained by the banner.
